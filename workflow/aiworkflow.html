B2B AI 禮品選購平台 - 功能邏輯規格書 (Functional Logic Specification) 資深技術專案經理(TPM) 2026年2月3日 1. 使用者故事與驗收標準(User Stories & AC) F01: AI禮品媒合功能 - User Story: 作為企業客戶,我想要輸入一段需求描述,以便讓AI幫我快速篩選出符合預算與場合的禮品。 - Acceptance Criteria (AC): - 系統必須支援自然語言描述輸入(如:預算500元、送給科技業客戶的伴手禮)。 - 媒合結果必須回傳至少一筆符合條件的商品清單,資訊需正確無誤。 - 失敗處理: 若AI無法識別或無匹配商品,應顯示「查無適合商品」之引導訊息,不可發生系統錯誤(Internal Server Error)。 - 搜尋反應時間應優化,若處理超過2秒需顯示Loading 載入狀態。 F02: 樣品合成預覽 - User Story: 作為企業客戶,我想要在上傳企業 Logo 後即時預覽合成效果,以便確認禮品視覺品質再決定下單。 - Acceptance Criteria (AC): - Logo 合成位置需精準,不得產生拉伸、扭曲或溢出邊界的情形。 - 合成圖片必須正常顯示,不得出現破圖或系統錯誤訊息。 - 系統應自動處理透明背景(PNG)之Logo,確保合成效果自然。 F03: 後台訂單管理系統 - User Story: 作為後台管理員,我想要在畫面上即時看到客戶的下單資訊,以便進行後續的業務開發與聯繫。 - Acceptance Criteria (AC): - 下單完成後,後台列表應即時更新顯示資料。 - 資訊必須完整包含: 客戶聯絡資訊、商品名稱、合成預覽圖連結。 - 確保資料傳輸正確性,後台顯示資訊需與前台客戶填寫完全一致。 2. 關鍵商業邏輯 (Business Rules) - 限制條件: - 合成上限: 同一使用者(以IP或User ID 識別)每日最多僅能進行100次合成操作。 - 檔案大小: 上傳之 Logo 檔案限制需小於5MB,僅支援JPG/PNG格式。 - 狀態流轉 (State Machine): - 訂單流程: 待確認→已接單→已完成。 - 狀態流轉具備「不可逆性」,一旦狀態推進則不可回溯。 3. 完整操作流程 (User Flow Path) Happy Path (成功路徑) Step1: 客戶在前台首頁輸入需求描述。 Step2: 系統顯示AI媒合後的推薦商品清單。 Step3: 客戶挑選商品並點選「合成預覽」。 Step4: 確認預覽效果滿意後,點選下單並填寫聯絡人資訊。 Step5: 後台接收到包含商品、合成圖與客戶資料的訂單。 Step6: 業務人員依據系統資訊主動聯繫客戶。 Edge Cases (失敗/例外路徑) - 連線中斷或錯誤: 若使用者在上述任何步驟發生錯誤或斷線,系統應自動彈出引導對話框,請客戶留下聯絡資訊,由專人主動聯繫以避免商機流失。 4. 待確認問題 (Open Questions) - AI媒合精準度: 當資料庫無完全匹配商品時,AI推薦的彈性範圍(Fuzzy Match)為何? - Logo 合成權限: 是否允許使用者在畫面上手動微調 Logo 位置或大小? - 驗證機制: 下單前是否需要手機簡訊(OTP) 驗證以過濾惡意下單?
